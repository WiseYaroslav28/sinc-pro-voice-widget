# 📖 Навык создания README: Руководство и стандарты

Этот навык предназначен для систематизации стандартов документирования проектов на GitHub. Он определяет, как создавать README-файлы, которые мгновенно вовлекают пользователя, объясняют пользу проекта простым языком и структурируют сложные технические детали.

---

## 🌟 1. Золотые правила проектирования README

Главное правило: **README пишется для людей, а не для компиляторов.**

### Слоеная структура (The Layered Approach)
Документ должен быть разделен на два смысловых слоя:
1.  **Пользовательский слой (сверху):** Фокусируется на пользе, визуале и быстром старте. Должен быть понятен человеку без технического образования.
2.  **Технический слой (снизу):** Описание архитектуры, стека, баз данных и алгоритмов. Скрывается под спойлеры (`<details>`) или выносится в отдельный файл `ARCHITECTURE.md`.

---

## 📐 2. Стандартная структура идеального README

### 1. Hero-секция (Первое впечатление)
*   **Эмодзи или Логотип:** Визуальный якорь в самом верху.
*   **Понятный слоган:** Одно-два предложения, объясняющие, какую именно проблему решает приложение.
*   **Флаги переключения языков:** Ссылки-якоря в первой строке для многоязычных проектов:
    ```markdown
    [🇷🇺 Русский](#ru) | [🇺🇸 English](#en)
    ```

### 2. Ценность для пользователя (Value Proposition)
Описывайте функции через **сценарии использования** (Use Cases).
*   *Плохо:* «Встроенный модуль OCR на базе WinRT API».
*   *Хорошо:* «**⛶ Перевод текста с картинок и экранов:** выделите область экрана рамкой, и перевод появится прямо поверх текста».
*   Выделите 3–5 ключевых преимуществ с использованием маркированных списков и жирного шрифта.

### 3. Быстрый старт (Quick Start)
Не заставляйте пользователя читать многостраничные инструкции перед запуском. Дайте ему схему «Запуск за 3 шага»:
1.  Скачать/Установить.
2.  Запустить.
3.  Нажать горячую клавишу / выполнить команду для проверки.

### 4. Визуализация (Showcase)
Люди любят глазами. Обязательно вставляйте:
*   Скриншоты интерфейса.
*   Анимированные GIF-файлы, демонстрирующие основной процесс работы (запись экрана).
*   Аудио или видео файлы, если проект работает со звуком.

### 5. Интерактивный гид / Таблица хоткеев
Если у программы много функций, оформите их в виде аккуратных таблиц Markdown. Таблицы легко сканируются глазами.
Пример таблицы горячих клавиш:
```markdown
| Горячие клавиши | Действие |
| :--- | :--- |
| `Ctrl + Shift` | Озвучить текст |
```

### 6. Секция «Под капотом» (Under the Hood)
Используйте тег `<details>`, чтобы скрыть сложные технические данные:
```html
<details>
<summary>🛠️ Технические детали реализации для разработчиков</summary>

### Технологический стек:
* Стек технологий...
### Алгоритмы:
* Описание алгоритмов...

</details>
```

## 🎨 3. Использование GitHub Markdown

Используйте встроенные фичи GitHub для улучшения читаемости:

### 1. Блоки внимания (GitHub Alerts)
Используйте их для акцентирования внимания:
```markdown
> [!NOTE]
> Важная информация, которую нужно знать.

> [!IMPORTANT]
> Критически важное требование для работы программы.

> [!WARNING]
> Предупреждение о возможных сбоях или ограничениях.
```

### 2. Спойлеры для объемных логов или списков
Если нужно приложить пример конфига или логов, прячьте их под `<details>`.

---

## 🌐 4. Особенности двуязычных (Многоязычных) README

Если репозиторий ориентирован на международную и локальную аудитории:
*   Используйте локальные ссылки-якоря (`<a id="ru"></a>` и `<a id="en"></a>`) для быстрого перехода.
*   Держите переводы синхронизированными. Если добавилась фича в русском описании, она должна появиться и в английском.
*   Используйте локализованные эмодзи-флаги для визуального разделения разделов.
