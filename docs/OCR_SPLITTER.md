# Правила локальной сегментации предложений (OCR Splitter)

В данном документе зафиксированы выводы, алгоритмы и правила, выработанные в процессе оптимизации локального (оффлайн) сплиттера предложений на бэкенде Rust ([lib.rs](file:///c:/Antigravity%20projects/voice-server/src-tauri/src/lib.rs)).

## Ограничения и особенности локального режима (без ИИ)
При работе без Gemini API (в режиме `ocr_mode == "local"`) сплиттер опирается исключительно на геометрические метрики распознанных Tesseract OCR слов (координаты Bounding Box) и грамматические эвристики. Основные вызовы:
1. Нестабильность координат букв в OCR.
2. Неполные предложения (пропущенные знаки препинания).
3. Специфические короткие строки (логи вроде `Worked for 10s ->`, пункты меню, заголовки), которые не должны сливаться с основным текстом.
4. Ошибки распознавания иконок (например, 📖 распознается как буква `G` или `O`).

## Золотые правила сегментации (Golden Rules)

### 1. Выравнивание по началу содержательного текста (`text_start_x`)
* **Проблема:** Строки списков начинаются с маркеров (например, `1. `, `* `, `• `, `- `). Сравнение крайних левых координат (`min_x`) приводит к ложным сдвигам и разбиению абзаца, так как строка без маркера начинается левее.
* **Решение:** Введена функция `is_list_marker(text: &str) -> bool`. Если строка начинается с маркера, координата выравнивания `text_start_x` берется по координате X второго слова. Если маркера нет — по первому.

### 2. Защита от невидимых символов на концах строк (`clean_ends_with_terminal`)
* **Проблема:** Tesseract OCR часто добавляет невидимый мусор, управляющие символы или zero-width space (`\u{200b}`) после знаков препинания (например, после `>` в логе `Worked for 10s ->`). Стандартный метод `.ends_with('>')` в Rust возвращает `false`, вызывая ложное слияние со следующей строкой.
* **Решение:** Создана функция `clean_ends_with_terminal(text: &str) -> bool`, которая очищает строку от пробелов, управляющих символов и `\u{200b}` перед поиском терминальных спецсимволов (`.`, `?`, `!`, `:`, `;`, `>`).

### 3. Фильтрация коротких строк по высоте шрифта (`is_prev_line_long`)
* **Проблема:** В небольших выделенных областях короткая строка (например, `Worked for 10s`) может составлять более 75% от максимальной ширины строки в этой области. Это приводило к тому, что строка считалась "длинной" и сливалась со следующей строкой.
* **Решение:** Добавлен критерий минимальной физической ширины строки относительно высоты шрифта:
  ```rust
  let is_prev_line_long = prev_line.width >= max_line_width * 0.75 
      && prev_line.width >= prev_line.avg_word_height * 12.0;
  ```
  Если ширина строки меньше 12 высот шрифта, она считается гарантированно короткой (лог, заголовок) и слияние блокируется.

### 4. Динамический масштаб высоты пары (`pair_h`)
* **Проблема:** Локальные занижения высоты букв (например, из-за точек в коде или подчеркиваний) занижали геометрические пороги сплиттера, ложно разделяя строки.
* **Решение:** Для всех расчетов используется максимальная высота шрифта пары строк:
  ```rust
  let pair_h = prev_line.avg_word_height.max(curr_line.avg_word_height).max(1.0);
  ```

### 5. Приоритет грамматики над геометрией
* **Продолжение со строчной буквы:** Если строка начинается со строчной (маленькой) буквы или открывающей скобки/кавычки (`is_lowercase_start || starts_with_bracket`), она считается грамматическим продолжением (`is_continuation == true`). Это отменяет мягкие геометрические разделения (сдвиг по X, изменение шрифта) и склеивает строки.
* **Начало с заглавной буквы:** Если строка начинается с заглавной буквы, слияние по потоку происходит только при условии, что предыдущая строка длинная (`is_prev_line_long == true`) и не закончилась терминальным знаком. Любые сдвиги по X или смена шрифта приведут к разделению.

### 6. Обработка мусорных одиночных букв (Иконки)
* **Проблема:** Иконки часто распознаются как одиночные заглавные буквы (например, `G`, `O`), что ломает определение регистра начала строки.
* **Решение:** Создан белый список легитимных однобуквенных слов в начале предложений: `is_valid_single_word_start` (`Я`, `В`, `С`, `У`, `К`, `О`, `А`, `И`, `I`, `A`). Если первый символ строки состоит из 1 буквы и его нет в белом списке, он признается мусором OCR/иконкой, и регистр начала строки проверяется по второму слову.

## От чего мы отказались (велосипеды, которые не работают)
* **Жесткая привязка к `last_h`:** Привязка к высоте последнего слова предыдущей строки давала сбои, если строка заканчивалась на точку или короткий символ. Использование `pair_h` решает эту проблему.
* **Слияние без проверки регистра начала следующей строки:** Приводило к склеиванию заголовков и пунктов меню с последующим текстом.
* **Сравнение крайних левых координат (`min_x`) для списков:** Приводило к "каскадному эффекту", когда строки одного пункта списка считались сдвинутыми и разбивались на отдельные предложения. Использование `text_start_x` решает эту проблему.
