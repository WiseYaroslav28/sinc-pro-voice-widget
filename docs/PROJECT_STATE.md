# Реестр Состояний SINC PRO

Этот документ описывает актуальное состояние проекта, принятые технические решения и архитектуру. **ИИ-Агенты обязаны сверяться с этим файлом перед любыми изменениями.**

## 🏗 Архитектура

Проект построен на связке:
* **Бэкенд (Rust / Tauri)**: Отвечает за системные хуки клавиатуры (WinAPI), работу с нативным буфером обмена (Save-Copy-Restore), управление DPI мониторов, хеширование токенов (Edge TTS) и сетевые запросы к ИИ-моделям (Gemini) в обход CORS.
* **Фронтенд (HTML/CSS/JS)**: Vanilla JS. Стилизация по дизайн-системе Stitch без тяжелых фреймворков. Использование принципа переиспользования компонентов (DRY).

## 🛠 Принятые Технические Решения (Golden Rules)
1. **Переиспользование компонентов**: Никакого копипаста (DRY). Использовать существующие UI-элементы для новых разделов (продублировать существующую панель TTS).
2. **Zero Trust Policy**: Не доверять сообщениям в старых коммитах, пока код не проверен вручную. Однако, ЗАПРЕЩАЕТСЯ "отменять" верификацию того, что пользователь уже подтвердил (`[VERIFIED]`).
3. **WinAPI vs Web APIs**: Из-за ограничений WebView критически важные системные взаимодействия (буфер, курсор, хуки клавиатуры, анти-залипание) реализованы строго на Rust-бэкенде.
4. **Безопасность буфера**: Ни одна операция автоматического чтения выделения (`Ctrl+C`) не должна уничтожать текст, скопированный пользователем ранее.
5. **Сохранение контента**: Все записанные сессии диктовки сбрасываются в локальную историю JSON, страхуя пользователя от сбоев API. Сброс по Escape имеет 5-секундный таймер.

## 🐛 Известные баги и подводные камни (Gotchas)
Чтобы не тратить время на повторные исследования, соблюдайте эти доказанные на практике правила для данного проекта:

1. **Tailwind JIT и Произвольные классы**: Сборщик Tailwind в этом проекте **не компилирует** на лету произвольные (arbitrary) значения, добавленные динамически или редко используемые (например, `w-[40px]`, `bottom-[30px]`, `group-hover:pointer-events-auto`).
   * **Решение**: Если нужен точный отступ, размер или специфический цвет, которого точно нет в базовой сборке, **используйте жесткие inline-стили** (`style="width: 40px; bottom: 30px;"`). Иначе элемент просто "схлопнется" в ноль и будет невидим/неработоспособен.
2. **Сброс аудио при снятии с паузы**: Браузерный HTML5 `<audio>` элемент **сбрасывает воспроизведение в 0:00**, если вы повторно присваиваете ему тот же самый `src` (даже если строка идентична).
   * **Решение**: Перед загрузкой аудио всегда проверяйте кастомный атрибут (`audio.dataset.loadedIndex === index.toString()`). Если индекс совпадает, просто вызывайте `audio.play()`, не трогая `src`.
3. **Hover-эффекты и пропадающие меню**: Из-за отсутствия компиляции `group-hover:pointer-events-auto`, всплывающие меню могут оставаться `pointer-events-none` навсегда, из-за чего мышка проваливается сквозь них.
   * **Решение**: Откажитесь от переключения `pointer-events-none/auto` по hover. Делайте меню всегда кликабельным (оно всё равно невидимо, если скрыто через JS класс `hidden` когда фича выключена). Если нужен мостик для hover, используйте прозрачный `div` с жестко заданным `style` и `z-index`, перекрывающим только зазор, а не саму кнопку.
4. **Сброс безрамочного оформления и появление нативного Titlebar/белой рамки Windows 11 при потере фокуса или скриншотах**:
   * **Проблема**: На Windows 11 DWM (Desktop Window Manager) при потере фокуса с окна или вызове системных утилит (например, «Ножницы») принудительно перерисовывает неклиентскую область окна (Titlebar). Если регион клика/отрисовки окна (`SetWindowRgn`) захватывает верхнюю границу физического окна (координаты `y < 30px`), DWM считает, что там находится стандартная область заголовка, и рисует нативную рамку Windows с кнопками управления, сдвигая и обрезая контент WebView2.
   * **Решение**: Единственное технически рабочее решение для Windows 11 DWM — это полное физическое исключение области заголовка окна из региона отрисовки ОС через `SetWindowRgn` в сочетании со сдвигом контента вниз и WinAPI-инициализацией при старте:
     1. Фиксировать физические размеры окон в `tauri.conf.json` на максимальных значениях (например, `560x380` для виджета TTS) и переключать только регионы кликов (`SetWindowRgn`) для имитации изменения размера.
     2. Сдвигать интерактивный контент виджета вниз в HTML/CSS с помощью отступа `padding-top: 40px`.
     3. Настраивать регион кликов `set_click_region` так, чтобы верхняя граница начиналась с `y = 35px` (или ниже). Это надежно исключает область заголовка окна (координаты `y = 0..30px`) из отрисовки DWM, предотвращая появление нативной рамки.
     4. Вызывать бэкенд-команду `resize_bottom_up_phys` при старте приложения (выполняет WinAPI `SetWindowPos` с флагами сброса Z-порядка и активации), чтобы сбросить DWM-оформление и зафиксировать безрамочный режим.
   * **Почему НЕ сработают другие попытки (Важно для будущей разработки!)**:
     * *Попытка отключить тени (`"shadow": false`) или декорации (`"decorations": false`) в конфиге*: Не помогает, так как Windows 11 принудительно восстанавливает Titlebar при потере фокуса или вызове «Ножниц», если регион отрисовки окна доходит до верхнего края (`y < 30px`).
     * *Попытка сделать мелкие отступы региона клика со всех сторон (например, `3px` со всех краев без сдвига контента вниз)*: Не помогает, так как при `y = 3px` верхняя граница окна всё равно попадает в зону ответственности DWM для заголовка. Заголовок прорисуется поверх контента.
     * *Попытка создавать окно скрытым (`"visible": false`) и показывать его позже через `.show()`*: Ломает инициализацию DWM-эффектов для нересайзимых окон на Windows 11. Окно при потере фокуса гарантированно получит белую рамку. Окно ОБЯЗАНО создаваться как `visible: true` и моментально скрываться в JS до отрисовки первого кадра.
     * *Попытка менять физический размер окна через `setSize` / `resize_window`*: Любой ресайз окна в рантайме на Windows 11 сбрасывает DWM-атрибуты и возвращает белую рамку.
