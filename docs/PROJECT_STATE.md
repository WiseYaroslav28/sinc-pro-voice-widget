# Реестр Состояний SINC PRO

Этот документ описывает актуальное состояние проекта, принятые технические решения и архитектуру. **ИИ-Агенты обязаны сверяться с этим файлом перед любыми изменениями.**

## 🏗 Архитектура

Проект построен на связке:
* **Бэкенд (Rust / Tauri)**: Отвечает за системные хуки клавиатуры (WinAPI), работу с нативным буфером обмена (Save-Copy-Restore), управление DPI мониторов, хеширование токенов (Edge TTS) и сетевые запросы к ИИ-моделям (Gemini) в обход CORS.
* **Фронтенд (HTML/CSS/JS)**: Vanilla JS. Стилизация по дизайн-системе Stitch без тяжелых фреймворков. Использование принципа переиспользования компонентов (DRY).

## 🛠 Принятые Технические Решения (Golden Rules)

1. **Переиспользование компонентов**: Никакого копипаста (DRY).
2. **Безопасность буфера**: Ни одна операция автоматического чтения выделения (`Ctrl+C`) не должна уничтожать текст, скопированный пользователем ранее.
3. **Бесшовная анимация и клик-регионы**:
   * Для предотвращения обрезки краев капсулы во время сжатия применены два параллельных решения:
     * **CSS-ускорение скрытия**: При удалении класса `.expanded` внутреннее содержимое капсулы (кнопки, таймер) мгновенно исчезает за **100 мс**, а сама оболочка капсулы плавно сужается к центру за **350 мс**. Это предотвращает визуальное "сжатие обрубленного контента".
     * **Задержка клик-региона Tauri**: Физическое сужение кликабельного региона ОС (функция `updateClickRegion`) откладывается на **450 мс** через `setTimeout` с защитной проверкой `if (!isMouseOver)`. Это гарантирует, что Windows 11 не обрежет края капсулы до того, как она полностью сожмется в маленький кружок `56x56px`.
4. **Управление слоями (Z-Index)**:
   * Оболочка `#active-pill` позиционирована относительно (`position: relative`) и поднята на `z-index: 30`, чтобы перекрывать невидимый ховер-мост веера пресетов (`z-index: 20`). Это возвращает кликабельность кнопке «Отправить» (галочке).
   * Всплывающие баннеры (отмена, ошибки) вынесены на уровень `z-index: 40`, чтобы они полностью перекрывали капсулу сверху и были 100% интерактивны.
5. **Синхронизация иконок**: Все переключатели в шапке дашборда используют те же иконки, что и сами плавающие виджеты или соответствующие разделы бокового меню (например, `mic` для капсулы, `record_voice_over` для озвучки).
6. **Локальная сегментация предложений (OCR Splitter)**: При объединении распознанных строк бэкенд должен учитывать очистку терминалов от невидимых символов (`clean_ends_with_terminal`), выравнивание списков по второму слову (`text_start_x`), а также защищать короткие строки логов по высоте шрифта (порог 12 высот букв). См. подробности в [OCR_SPLITTER.md](file:///c:/Antigravity%20projects/voice-server/docs/OCR_SPLITTER.md).
7. **Кнопка мгновенной отмены ИИ-запроса (в режиме отправки)**: Для предотвращения случайного отображения кнопки `#btn-cancel-sending` (крестик отмены ИИ) вне состояния отправки, она отображается только при совпадении двух условий: наличие класса `.visible` (активен только в состоянии `sending`) и ховер на родительский враппер `.active-pill-wrapper:hover #btn-cancel-sending.visible` в CSS. При клике на кнопку флаг `currentRequestCancelled` на фронтенде отсекает любые запоздалые сетевые ответы или ошибки, а вызов команды `cancel_active_request` останавливает Gemini-запросы на бэкенде через атомарный флаг.
8. **3-позиционная схема управления**:
   * **Выбор UI**: Отвергнуты классические выпадающие списки (слишком громоздко) и чекбоксы (не позволяют переключать 3 состояния). Выбран кастомный горизонтальный 3-позиционный слайдер («Выкл» / «Только хоткеи» / «Все включено»), который идеально вписался в дизайн-систему без усложнения интерфейса.
   * **Логика временного показа по хоткеям**: В режиме «Только хоткеи» (состояние `1`) плавающие виджеты остаются невидимыми. При активации хоткея они временно отображаются на экране для визуального подтверждения операции и плавно скрываются с задержкой (1.5–3 секунды) после завершения процесса (например, проигрывания TTS или распознавания OCR).
9. **Сохранение геометрии окон и мультимониторность**:
   * **Событийная модель сохранения**: Координаты X/Y и размеры окон сохраняются в `localStorage` при возникновении событий перетаскивания (`tauri://drag`) или изменения размеров (`tauri://resize`). Чтобы не перегружать I/O, для главного окна применен дебаунс (300 мс).
   * **Защита от смены конфигурации мониторов**: Отвергнуто слепое восстановление координат, так как при отключении внешнего монитора окна «улетали» за пределы видимости. На старте каждого окна выполняется проверка с помощью `availableMonitors()`. Если сохраненные координаты не попадают в охват ни одного из активных экранов, окно принудительно переносится в дефолтное безопасное положение на основном мониторе.
10. **Автозапуск Windows с флагом `--minimized`**:
    * **Логика запуска**: Автозапуск регистрируется в реестре `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`. При старте с аргументом `--minimized` (определяемым на бэкенде через `std::env::args()`), главное окно (дашборд) не показывается и прячется в системный трей.
    * **Восстановление состояния виджетов**: При автозапуске активные виджеты (Капсула, Озвучка, OCR в режиме `2` — «Все включено») автоматически считывают состояние и появляются на экране на своих местах. Дашборд при этом остается скрыт. Если приложение закрыли в состоянии, когда дашборд был открыт, при обычном запуске (без флага) дашборд восстанавливает видимость.
11. **Сохранение буфера обмена с изображениями и текстом**:
    * Реализована отказоустойчивая схема бэкапа пользовательского буфера через `backup_clipboard` и `restore_clipboard` на Rust.
    * Чтобы сохранялись картинки, в список разрешенных форматов включены `CF_DIB` (8) и `CF_DIBV5` (17) — стандартные побайтовые форматы растровых изображений Windows.
    * Для защиты от блокировки буфера операционной системой применен цикл повторных попыток (`retry-loop` до 15 попыток с интервалом в 15 мс) при вызове `OpenClipboard`.
    * Если бэкап пустой (например, буфер обмена изначально был пуст или бэкап завершился с ошибкой), функция восстановления не очищает буфер в ноль, сохраняя новый текст авто-вставки.
12. **Взаимная приостановка записи и TTS**:
    * При старте озвучки (TTS) плеер генерирует глобальное событие `tts-playing-started`, которое ставит диктофонную запись на паузу, если она запущена.
    * При старте/продолжении записи в Капсуле генерируется событие `recording-started`, которое автоматически ставит воспроизведение TTS на паузу.
13. **Оптимизация префетча TTS (устранение заиканий и rate limit)**:
    * Внедрена строго последовательная очередь префетча предложений на фронтенде (JS, `VoiceCore`) в цикле `while (true)` с флагом `isNetworkFetching`, который предотвращает параллельный запуск нескольких сетевых запросов при переключении предложений.
    * Реализована мгновенная массовая параллельная предзагрузка всех кэшированных предложений из IndexedDB на шкалу прогресса при вызове `loadText` / `appendText` (метод `loadCachedSentences`), что исключает их ожидание в последовательной очереди префетча.
    * Размер буферизации для сетевых запросов ограничен окном в 4 предложения: текущее + 3 предложения наперед. Если все они загружены, очередь "засыпает". Если предложения в окне находятся в процессе загрузки, очередь ожидает их завершения, не прерывая цикл.
    * На бэкенде в Rust (`lib.rs`) реализован глобальный асинхронный мьютекс `EDGE_TTS_MUTEX`, гарантирующий, что все запросы к Edge TTS обрабатываются строго один за другим. Это полностью устраняет rate-limiting со стороны Microsoft.
    * Все сетевые вызовы в Rust обернуты в `tokio::time::timeout` с лимитом в 10 секунд для исключения вечных зависаний очереди префетча при нестабильном интернет-соединении.
    * Кэширование предложений в IndexedDB проверяется мгновенно без сетевых пауз и задержек, а для первого сетевого запроса задержка 500 мс сбрасывается в ноль для мгновенного старта.
14. **Интеграция автообновлений (Tauri Updater)**:
    * Отказались от стандартных системных диалогов Tauri (`"dialog": false` по умолчанию), так как они блокируют интерфейс и не дают контролировать процесс. Процесс проверки и установки перенесен на JS-сторону фронтенда через глобальные API `window.__TAURI__.updater` и `window.__TAURI__.process`.
    * Реализован кастомный прогресс-бар в панели настроек с отображением прогресса загрузки в процентах и мегабайтах через асинхронный коллбэк метода `downloadAndInstall`.
    * Настроено обязательное локальное подписание релизов (`tauri signer`) с шифрованием приватного ключа. Папка `.tauri-keys/` добавлена в `.gitignore` для исключения утечек.

## 🐛 Известные баги и подводные камни (Gotchas)

1. **Tailwind JIT и Произвольные классы**: Сборщик Tailwind **не компилирует** на лету динамически добавленные произвольные классы (например, `w-[40px]`). Используйте жесткие inline-стили (`style="width: 40px;"`).
2. **Сброс аудио при снятии с паузы**: Браузерный HTML5 `<audio>` сбрасывает воспроизведение в 0:00 при повторном присвоении идентичного `src`. Всегда проверяйте `audio.dataset.loadedIndex` перед изменением `src`.
3. **Сброс безрамочного оформления и Titlebar Windows 11**:
   * DWM Windows 11 восстанавливает системную белую рамку окна при потере фокуса, если регион отрисовки окна (`SetWindowRgn`) заходит в зону `y < 30px` (Titlebar).
   * **Решение**: Верхняя граница клик-региона `set_click_region` должна начинаться строго с `y = 30px` (или ниже). Контент сдвигается вниз с помощью `padding-top` не менее `32px`.
4. **Затирание настроек при межоконной синхронизации (TTS)**: 
   * Передача частичных настроек (например, только измененной скорости `{ speed: 1.2 }` из виджета) через событие `tts-state-sync` полностью перезаписывала объект `this.settings` в `VoiceCore` во втором окне, затирая поле `voice` в `undefined`. Это приводило к ошибке Tauri `missing required key voice` и блокировке вызова озвучки.
   * **Решение**: В [voice-core.js](file:///c:/Antigravity%20projects/voice-server/src/voice-core.js) логика перезаписи заменена на слияние через spread-оператор: `this.settings = { ...this.settings, ...p.settings }`.
5. **Неподдерживаемые ИИ-модели для ввода аудио/изображений (STT/OCR)**: 
   * Выбор пользователем модели-генератора речи `"gemini-3.1-flash-tts-preview"` для диктовки или распознавания экрана приводил к падению запросов с ошибкой `400 Bad Request` («Audio input modality is not enabled for this model»).
   * **Решение**: На стороне Rust-бэкенда в [lib.rs](file:///c:/Antigravity%20projects/voice-server/src-tauri/src/lib.rs) добавлена автоматическая подмена (заплатка) модели на `"gemini-2.0-flash"`, если имя выбранной модели содержит `"tts-preview"`.
6. **Ошибка ERR_CONNECTION_REFUSED при запуске Tauri debug-версии агентом**:
   * Debug-сборка Tauri-приложения привязывается к dev-серверу Node.js. Запуск исполняемого `.exe` файла напрямую из фоновой консоли ИИ-агента без запущенного Node-сервера вызывает ошибку `ERR_CONNECTION_REFUSED` в WebView2. Для корректной отладки необходимо запускать dev-окружение командой `npm run tauri dev`.
   * Поскольку фоновые консольные сессии ИИ-агентов изолированы от GUI, для физического запуска окна на мониторе пользователя используется планировщик Windows (`schtasks` / `Register-ScheduledTask`) с интерактивным типом входа (`LogonType Interactive`).

## 🔮 Направления разработки на будущее

### Механизм автоматических обновлений (Tauri Updater)
Для удобства распространения новых версий (таких как `3.4.2`) планируется реализовать встроенный механизм скачивания обновлений:
1. **Конфигурация Tauri**:
   * Включить плагин updater в `tauri.conf.json`:
     ```json
     "plugins": {
       "updater": {
         "active": true,
         "endpoints": [
           "https://raw.githubusercontent.com/WiseYaroslav28/sinc-pro-voice-widget/master/docs/updater.json"
         ],
         "pubkey": "ЗДЕСЬ_БУДЕТ_ПУБЛИЧНЫЙ_КЛЮЧ_ПОДПИСИ"
       }
     }
     ```
2. **Генерация ключей подписи**:
   * Разработчик использует команду `tauri signer generate` для создания пары ключей. Приватный ключ хранится в секретах сборки (например, на GitHub Actions), а публичный прописывается в конфиге.
3. **Сборка релизов**:
   * При компиляции релиза Tauri автоматически генерирует архив обновления `.zip` / `.tar.gz` и сигнатуру подписи.
4. **updater.json**:
   * В корне репозитория (или в `docs/updater.json`) поддерживается JSON-файл следующего формата:
     ```json
     {
       "version": "3.4.2",
       "notes": "Описание релиза 3.4.2",
       "pub_date": "2026-06-07T05:30:00Z",
       "platforms": {
     ```
